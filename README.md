# AI Oppose AI Skill

一个用于审查 AI 输出的反对意见校准 Skill。

它的目标不是让 AI 为了反对而反对，而是帮助 AI 在合适的时候提出合理异议：当回答存在事实不确定、证据不足、目标偏移、风险不对称、关键反例缺失、过度自信、过度顺从或安全边界问题时，给出可检查、可分级、可执行的反对意见。

## 项目简介

`AI Oppose AI Skill` 将 AI 作为一个审查型 agent 使用，用来判断另一个 AI 的回答、计划、prompt、agent workflow、代码变更说明或其他生成内容是否需要被质疑。

它关注的问题包括：

* 当前回答是否偏离用户真实目标
* 事实前提是否可靠
* 证据是否足以支撑结论
* 是否遗漏关键风险或反例
* 反对强度是否与风险等级匹配
* 是否存在更稳、更小干预的替代路径
* 是否存在 prompt injection、敏感信息泄露或安全边界问题

最终决策仍然保留给用户或主 agent。这个 Skill 只提供审查、校准和决策支持。

## 适用场景

适合在以下场景中使用：

* 审查 AI 生成的回答是否过度顺从
* 审查复杂计划是否存在隐藏假设
* 检查 prompt 或 agent workflow 是否容易被注入攻击影响
* 判断某个建议是否证据不足或过度确定
* 在高风险任务前补充反例、风险和验证条件
* 为主 agent 提供“是否应反对”的第二意见
* 审查代码变更说明、迁移计划、执行方案或自动化流程

## 不适用场景

不建议在以下低风险任务中默认触发：

* 普通翻译
* 简单润色
* 日常总结
* 开放式头脑风暴
* 用户没有要求审查、校准、风险判断或反对意见的普通问答

除非这些任务中出现明显事实风险、安全风险、隐私风险或目标偏移。

## 核心原则

这个 Skill 基于七个审查原则工作：

1. **目标一致**：检查回答是否只满足表面指令，却偏离用户真正要解决的问题。
2. **事实优先**：事实前提未验证或明显错误时，不在错误地基上继续推进。
3. **风险不对称**：风险越高、越不可逆，越需要提前提出反对。
4. **证据缺口**：信息不足以支持确定结论时，避免过早下判断。
5. **反例压力**：主动检查方案在什么情况下会失败。
6. **最小干预**：反对要分级，优先选择足以修复问题的最低强度。
7. **替代方案**：反对意见必须帮助任务继续推进，而不是单纯否定。

## Dissent Level

Skill 使用 `Dissent Level 0-4` 来校准反对强度：

| Level | Verdict        | 含义                                 |
| ----- | -------------- | ---------------------------------- |
| 0     | pass           | 无明显问题，不需要提出反对                      |
| 1     | pass           | 有轻微假设、遗漏或措辞风险，可轻提示                 |
| 2     | revise         | 存在明显漏洞、证据不足、目标偏移或中等风险，需要修改         |
| 3     | revise / block | 存在高风险、不可逆后果、严重误导或关键事实未验证，应暂停关键动作   |
| 4     | block          | 涉及违法、有害、隐私泄露、恶意绕过、安全破坏或不可接受风险，必须阻断 |

## 快速开始

克隆仓库：

```bash
git clone https://github.com/WooyoohL/AI-oppose-AI-skill.git
```

并将```SKILL.md```复制到全局/项目 ```.codex/skills/```目录下

推荐调用方式：

```text
使用 $ai-opposes-ai 审查下面这段 AI 输出。

用户原始请求：
[粘贴用户请求]

待审查的 AI 回答 / 计划 / prompt / workflow：
[粘贴待审查内容]

必要上下文：
[粘贴上下文]

输出偏好：
[轻量 / 完整 / 由你按风险选择]
```

## 输出格式

Skill 会根据风险等级选择轻量或完整输出。

轻量输出适合低风险场景，通常包含：

* Verdict
* Dissent Level
* 是否需要提出反对意见
* 一句话结论
* 简短依据
* 可选修改

完整输出适合中高风险场景，通常包含：

* 审查结论
* 判断依据
* 主要反对意见
* 风险与反例
* 建议改法
* 可直接使用的表达
* 反对意见自检
* 仍需人来判断的点


## 设计目标

这个项目追求的是审查质量，而不是反对数量。

好的反对意见应该满足：

* 不为了反对而反对
* 不把个人偏好包装成客观事实
* 不用道德评判替代证据分析
* 不因低风险小瑕疵阻断任务
* 不在证据不足时武断下结论
* 能指出风险，也能给出更稳路径
* 保留用户或主 agent 的最终选择权
